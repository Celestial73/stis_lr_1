# Предметная область: медицинская клиника

Фрагмент описывает типовой сценарий обслуживания пациента в амбулаторной медицинской клинике: **запись на приём → назначение услуг → оплата → оказание услуги врачом в кабинете**. Область выбрана для БДЗ-1 (семантическая сеть) и БДЗ-2 (фреймовая модель) как связный процесс с понятной иерархией понятий и несколькими типами отношений.

---

## Контекст и границы модели

Моделируется **не вся клиника целиком**, а один законченный бизнес-процесс визита:

- регистрация пациента на приём;
- формирование приёма из одного или нескольких назначений (строк приёма);
- привязка каждого назначения к конкретной медицинской услуге;
- оплата приёма;
- проведение приёма врачом в указанном кабинете.

**Вне scope** (намеренно упрощено): стационар, лабораторная диагностика как отдельный поток, страховые полисы, расписание смен, электронная медкарта, отмена и перенос приёма.

---

## Участники и роли

| Роль | Кто | Действия в модели |
|------|-----|-------------------|
| Пациент | физическое лицо, обращающееся в клинику | записывается на приём, оплачивает приём |
| Врач | медицинский работник | оказывает услугу по приёму в кабинете |
| Клиника (неявно) | организация | предоставляет услуги, кабинеты, приём как сущность учёта |

Оба активных участника обобщены понятием **УЧАСТНИК** (ISA-иерархия).

---

## Основной сценарий (по шагам)

1. **Пациент** инициирует **ЗАПИСЫВАЕТ(пациент, приём)** — создаётся сущность **ПРИЁМ** (запись на посещение).
2. К приёму добавляются **НАЗНАЧЕНИЕ** — строки, каждая фиксирует выбранную **УСЛУГУ** (**СОДЕРЖИТ**, **УКАЗЫВАЕТ_НА**).
3. Если услуга — **КОНСУЛЬТАЦИЯ**, она наследует свойства общего типа **УСЛУГА** (ISA).
4. Пациент производит **ОПЛАТА** приёма (**ОПЛАЧИВАЕТ**).
5. **Врач** проводит **ОКАЗАНИЕ** услуги по этому приёму в конкретном **КАБИНЕТ** (**ОКАЗЫВАЕТ**).

Пример экземпляров: пациент «Иванов», приём «Приём_001», назначение «Назнач_001», услуга «Консультация_терапевта», оплата «Оплата_001», врач «Доктор_Сидоров», кабинет «Кабинет_12».

---

## Понятия (10)

| Понятие | Смысл |
|---------|--------|
| **УЧАСТНИК** | обобщённый субъект системы клиники |
| **ПАЦИЕНТ** | участник, записывающийся на приём и оплачивающий услуги |
| **ВРАЧ** | участник, оказывающий медицинские услуги |
| **УСЛУГА** | медицинская услуга клиники (абстрактный тип) |
| **КОНСУЛЬТАЦИЯ** | частный вид услуги (приём врача) |
| **ПРИЁМ** | запись пациента на посещение; контейнер назначений |
| **НАЗНАЧЕНИЕ** | строка приёма: ссылка на услугу и параметры (напр. доза) |
| **ОПЛАТА** | факт оплаты приёма |
| **ОКАЗАНИЕ** | процесс проведения услуги пациенту |
| **КАБИНЕТ** | помещение, где врач принимает пациента |

---

## ISA-иерархия

```
ПАЦИЕНТ      ──isa──► УЧАСТНИК
ВРАЧ         ──isa──► УЧАСТНИК
КОНСУЛЬТАЦИЯ ──isa──► УСЛУГА
```

Транзитивное замыкание **ISA+** совпадает с **ISA0** (цепочек наследования длиной > 1 нет).

---

## Отношения и предикаты

| Предикат | Смысл |
|----------|--------|
| `ЗАПИСЫВАЕТ(x, y)` | пациент x записывается на приём y |
| `СОДЕРЖИТ(y, p)` | приём y содержит назначение p |
| `УКАЗЫВАЕТ_НА(p, t)` | назначение p ссылается на услугу t |
| `ОПЛАЧИВАЕТ(x, y, o)` | пациент x оплачивает приём y платежом o |
| `ОКАЗЫВАЕТ(c, y, a)` | врач c оказывает услугу по приёму y в кабинете a |

Переменные полной сетевой модели:  
`x:ПАЦИЕНТ`, `c:ВРАЧ`, `y:ПРИЁМ`, `p:НАЗНАЧЕНИЕ`, `t:УСЛУГА`, `o:ОПЛАТА`, `a:КАБИНЕТ`.

---

## Атрибуты (фреймовая модель, БДЗ-2)

**Характеристические фреймы (H1–H5):**

| Объект | Атрибут | Домен | Пример значений |
|--------|---------|-------|-----------------|
| ПРИЁМ | статус | СТАТУС.ПРИЁМА | записан / оплачен / оказан |
| ОПЛАТА | сумма | СУММА | стоимость приёма |
| НАЗНАЧЕНИЕ | доза | ДОЗА | дозировка препарата или кратность |
| ОКАЗАНИЕ | кабинет | КАБИНЕТ | номер кабинета |
| УСЛУГА | тип | ТИП.УСЛУГИ | консультация / процедура / анализ |

**Событийные фреймы (F1–F4):** ЗАПИСЫВАЕТ, ДОБАВЛ.НАЗН., ОПЛАЧИВАЕТ, ОКАЗЫВАЕТ — связываются общими переменными `x`, `y`, `p`, `o`, `a`.

---

## Ограничения (инварианты домена)

- в каждом приёме — от 1 до 10 назначений;
- каждое назначение ссылается ровно на одну услугу, доза ≥ 1;
- каждое оказание связано ровно с одним врачом и одним кабинетом.

---

## Зачем выбран этот фрагмент

- **10 понятий** — в требуемом диапазоне 10–12 для задания.
- Есть **ISA**, **составные связи** (приём ↔ назначения) и **события** (запись, оплата, оказание).
- Один сценарий одинаково описывается **семантической сетью** (БДЗ-1) и **двухуровневой фреймовой моделью** (БДЗ-2).
- Достаточно богат для ER-диаграммы, полной и сокращённой сети, графика фреймов и формул с кванторами.

## Адаптация для ЛР-1 (система вывода с концептами)

Для лабораторной работы №1 модель **упрощена** до 6 концептов с **транзитивной** ISA-иерархией и реализована на **C#** (проект `LR_1.Clinic`).

### Отличия от БДЗ-модели

| Аспект | БДЗ (выше) | ЛР-1 |
|--------|------------|------|
| Понятий | 10 | **6** |
| ISA | 1 уровень | **3 уровня** (корень **Сущность**) |
| Связи | 5 | **2 обязательных** + 3 опциональных |
| Консультация | отдельный концепт | экземпляр **Услуга** (`тип = консультация`) |
| Назначение, Оплата, Оказание, Кабинет | отдельные понятия | атрибуты / факты в мирах Крипке |

### 6 концептов ЛР-1

```
Сущность
├── Участник
│   ├── Пациент
│   └── Врач
├── Услуга
└── Приём
```

### Связи ЛР-1

| Связь | Сигнатура |
|-------|-----------|
| **ЗАПИСЫВАЕТ** | Пациент × Приём |
| **ОКАЗЫВАЕТ** | Врач × Приём |
| **СОДЕРЖИТ** *(опц.)* | Приём × Услуга |
| **ОПЛАЧИВАЕТ** *(опц.)* | Пациент × Приём |

**ISA на связях:** `ЗАПИСЫВАЕТ`, `ОПЛАЧИВАЕТ` → `ДействиеПациента`; `ОКАЗЫВАЕТ` → `ДействиеВрача`.

### Экземпляры

| Id | Концепт | Пример слотов |
|----|---------|---------------|
| `patient_ivanov` | Пациент | имя=Иванов, полис=12345 |
| `doctor_sidorov` | Врач | имя=Доктор_Сидоров, специализация=терапевт |
| `service_consult` | Услуга | название=Консультация_терапевта, тип=консультация |
| `appt_001` | Приём | статус, дата, кабинет=Кабинет_12 |

### Шкала Крипке (W0–W3)

| Мир | Факты | Статус приёма |
|-----|-------|---------------|
| W0 | — | — |
| W1 | ЗАПИСЫВАЕТ, СОДЕРЖИТ | записан |
| W2 | + ОПЛАЧИВАЕТ | оплачен |
| W3 | + ОКАЗЫВАЕТ | оказан |

### Запуск

Открыть `LR_1.sln` в Visual Studio → **F5** — консольная демонстрация instance-of, связей и миров Крипке.
